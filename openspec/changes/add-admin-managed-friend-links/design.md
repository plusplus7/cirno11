## Context

The React app currently renders public pages from build-time `SiteData`, with non-home routes using an `archive-aside` for profile links and archive stats. Admin content management is owner-only under `/plusplus7_admin_portal`, and array-like metadata such as photos and lab tools already follows a structured list pattern backed by `ContentStore`.

Content storage has two implementations: local files under `content/data/` for development and GitHub `data/` JSON files when repository-backed content is configured. Public releases are generated by `scripts/generate-public-data.ts`, so visitor-facing friend links must be included in generated static data rather than fetched from admin APIs.

## Goals / Non-Goals

**Goals:**

- Add a small friend-link section to the public aside that displays configured enabled links with avatar-sized icons and names.
- Let the owner add and delete friend links from the existing admin workspace.
- Store friend links through the existing content store abstraction so local and GitHub-backed modes behave the same way.
- Keep public rendering static and compatible with the current publish flow.

**Non-Goals:**

- Do not add public API fetching for friend links.
- Do not change the owner authentication/session model or admin route path.
- Do not add image upload support specifically for friend-link avatars; avatar icons are URL-based metadata.
- Do not redesign the whole public aside or admin workspace beyond the friend-link controls.

## Decisions

### Store friend links as first-class content metadata

Add a shared `FriendLink` type with `id`, `name`, `url`, `avatarUrl`, and `enabled`. Store arrays in `data/friend-links.json`, exposed through `ContentStore` as `listFriendLinks()` and `saveFriendLinks()`.

This mirrors `lab.json` and `photos.json`, keeps the feature independent from `about.json`, and avoids turning the about profile schema into a catch-all bucket.

Alternative considered: put friend links inside `AboutContent`. That would reduce one content-store method pair, but it couples profile prose and sidebar link management and makes structured admin controls harder to keep focused.

### Publish only enabled links

`generate-public-data.ts` should include `friendLinks` after filtering for `enabled`, while the admin API should return all friend links so hidden entries can still be managed.

This matches lab tool behavior and gives the owner a reversible hide action without deleting metadata.

Alternative considered: delete-only management with no enabled flag. That is simpler but makes temporary removals destructive.

### Use authenticated JSON endpoints

Add `GET /api/friend-links` and `PUT /api/friend-links` behind the existing owner-only middleware. Validate the incoming body with an `isFriendLinkArray` helper before saving.

This follows the current metadata API shape and avoids introducing item-level REST endpoints that the rest of the admin app does not use for photos or lab tools.

Alternative considered: `POST /api/friend-links` and `DELETE /api/friend-links/:id`. That is more granular, but inconsistent with the current save-whole-array metadata pattern.

### Reuse the structured metadata UI pattern

Add a friend-link admin tab or panel that uses the existing two-column manager layout: form fields on the left and a structured list on the right. The form should collect display name, avatar URL, target URL, and public visibility. Rows should show the avatar, name, URL, enabled/hidden state, and delete action.

This keeps owner workflows consistent with lab management and avoids raw JSON editing for routine updates.

Alternative considered: expose friend links through the generic JSON manager. That would be faster but conflicts with the requested dynamic add/delete workflow and the existing movement away from raw JSON for array metadata.

### Render links inside the existing aside

The public aside should gain an additional `aside-block` below the profile card/stats area on non-home routes. Each friend link should be a normal anchor using the stored URL, avatar image, and display name. External links should use safe new-tab behavior.

This preserves the existing home layout and public navigation while adding the requested sidebar column.

Alternative considered: add friend links to the home page or footer. That would make links more globally visible, but the request is specifically for the aside.

## Risks / Trade-offs

- Invalid or missing avatar URLs could create broken images in the aside → Use compact image sizing, empty `alt` text when the name is already visible, and validate required string fields before saving.
- Whole-array `PUT` can overwrite concurrent edits → This matches existing metadata flows; keep the feature consistent unless the admin app later moves to item-level APIs.
- Adding another admin tab can crowd the tab row on small screens → Rely on the existing wrapping/responsive tab layout and verify `/plusplus7_admin_portal` at mobile widths.
- Generated `site-data.json` schema changes can break strict consumers → Update shared `SiteData`, regenerate static data, and keep the fallback empty array when no friend-link JSON exists.

## Migration Plan

1. Add the new shared type, validation helper, and empty-array fallback for missing friend-link data.
2. Extend local and GitHub content stores to read and write `data/friend-links.json`.
3. Add authenticated admin API endpoints and client helpers.
4. Add admin structured friend-link management.
5. Include enabled links in generated `SiteData` and render the public aside section.
6. Run automated checks and manually verify public aside rendering plus admin add/delete persistence.

Rollback is straightforward: remove the friend-link UI/API/store additions and ignore `data/friend-links.json`. Existing content types and publish behavior remain unchanged.
